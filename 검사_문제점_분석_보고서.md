## 2. 종합 결론

현재 구현은 컴파일 가능한 상태이지만, 예약과 호텔·객실의 관계를 나타내는 안정적인 식별자가 없고, 변경 가능한 우편번호와 객실 번호를 관계 식별자이자 과거 내역 스냅샷으로 동시에 사용한다. 이 구조에서는 문서의 스냅샷 정책을 지켜도 문제가 생기고, 현재 구현처럼 일부 값을 연쇄 변경해도 다른 문제가 생긴다.

가장 먼저 해결해야 할 문제는 다음과 같다.

1. 호텔 우편번호·객실 번호 변경 후 활성 예약이 실제 호텔·객실과 분리되는 문제
2. 동일한 호텔명을 가진 다른 사장이 타인의 예약을 조회·처리할 수 있는 문제
3. 체크인·체크아웃 날짜를 놓치면 예약 상태가 영구적으로 멈추는 문제
4. 취소 승인 마감이 지난 뒤 다시 `CANCEL_PENDING` 상태를 만들 수 있는 문제
5. 기준 시간을 여러 달 진행할 때 중간 월 자동 처리가 누락되는 문제
6. 여러 파일 저장 중 일부만 성공하면 데이터 관계가 깨지는 문제

단순히 구현만 기획서 문구에 맞추면 해결되지 않는 항목도 있다. 특히 식별자와 스냅샷 정책, 놓친 체크인·체크아웃 처리, 패널티 종료일 의미, 자동 영업정지 취소 원인 보존은 기획서에 최소한의 정책 보완이 필요하다.

## 4. 치명적·높은 위험 로직 모순

### C-01. 변경 가능한 우편번호·객실 번호와 예약 스냅샷 정책의 구조적 충돌

**판정: 치명적, 현재 문서만으로는 일관된 구현 불가**

#### 근거

- 기획서 5.2와 6.4.2는 호텔·객실 정보가 변경되더라도 기존 예약의 과거 정보를 허용하고, 객실 변경 사항을 기존 `reservation.txt`에 적용하지 않는다고 설명한다.
- 설계서 4.3, 4.5, 7.1, 7.3은 예약의 `hotelName`, `postalCode`, `roomNumber`를 스냅샷으로 유지하며 활성 예약이 있어도 우편번호와 객실 번호 변경을 허용한다.
- 그러나 설계와 구현의 객실 식별 및 중복 검사는 `(postalCode, roomNumber)` 조합만 사용한다.
  - `src/main/java/service/GuestService.java:112-115`
  - `src/main/java/service/IntegrityService.java:137-142`
- 현재 구현은 호텔 우편번호 변경 시 예약·패널티·영업정지의 우편번호까지 연쇄 변경한다.
  - `src/main/java/service/HostService.java:99-129`
- 반면 객실 번호 변경 시 기존 예약의 객실 번호는 변경하지 않는다.
  - `src/main/java/service/HostService.java:132-137`

#### 모순 발생 시나리오

1. 우편번호 `11111`, 객실 `101`에 6월 20일 예약 A가 존재한다.
2. 사장이 객실 `101`의 번호를 `102`로 변경한다.
3. 예약 A는 스냅샷 정책에 따라 `101`을 계속 가진다.
4. 같은 실제 객실은 현재 `102`로 조회되므로, 동일 시간에 객실 `102` 예약 B를 생성할 수 있다.
5. 중복 검사는 A의 `101`과 B의 `102`가 다르다고 판단한다.

호텔 우편번호도 문서대로 스냅샷을 유지하면 같은 문제가 발생한다. 우편번호 변경 후 새 예약은 새 우편번호를 사용하므로, 변경 전 우편번호를 가진 활성 예약과 겹쳐도 다른 호텔로 판단된다.

현재 구현처럼 예약 우편번호를 연쇄 변경하면 중복 예약 위험 일부는 줄지만, 기획서·설계서가 요구하는 과거 내역 스냅샷을 훼손한다. 또한 객실 번호에는 같은 연쇄 변경을 적용하지 않아 정책도 일관되지 않다.

#### 영향

- 동일한 실제 객실에 중복 예약 가능
- 예약 변경, 패널티, 영업정지, 월별 평점 집계가 잘못된 호텔과 연결될 수 있음
- 변경 전 우편번호가 다른 호텔에서 재사용되면 과거 예약의 소유 관계가 왜곡될 수 있음
- 문서대로 구현해도, 현재 구현을 유지해도 결함이 남음

#### 권장 수정

- **기획서 최소 보완:** 호텔과 객실에는 변경되지 않는 내부 식별자가 존재하고, 표시용 이름·우편번호·객실 번호 스냅샷과 관계 식별자를 구분한다고 명시한다.
- **설계서:** `hotelId`, `roomId` 같은 안정적인 내부 키를 관계 판단에 사용하고, 스냅샷 필드는 출력과 과거 내역 보존에만 사용하도록 수정한다.
- **구현물:** 예약 중복, 소유권, 패널티, 영업정지, 월별 평점 집계를 내부 키 기준으로 변경한다.
- 내부 식별자 추가가 허용되지 않는 경우의 차선책은 활성 예약·패널티·영업정지가 남아 있는 동안 우편번호와 객실 번호 변경을 금지하는 것이다.

### C-02. 동일 호텔명 기준으로 타 사장 예약을 소유 예약으로 판정

**판정: 치명적, 구현 결함**

#### 근거

- 기획서 6.4.3은 현재 사장 ID가 소유한 호텔의 우편번호와 일치하는 예약만 조회하도록 규정한다.
- 구현은 예약 우편번호가 일치하거나 호텔명이 일치하면 소유 예약으로 판정한다.
  - `src/main/java/service/HostService.java:155-166`
  - `src/main/java/service/HostService.java:192-199`
- 호텔명은 고유값이 아니며, 기획서도 호텔명 중복을 금지하지 않는다.

#### 모순 발생 시나리오

1. 사장 A와 사장 B가 각각 이름이 `행복호텔`인 서로 다른 호텔을 등록한다.
2. 사장 A의 호텔 예약은 사장 B의 호텔 우편번호와 다르지만 호텔명이 같다.
3. 사장 B의 예약 관리 목록에 사장 A의 예약이 포함된다.
4. 해당 예약이 `CANCEL_PENDING`이면 사장 B가 승인 또는 거절할 수 있다.

#### 영향

- 다른 사장의 예약 정보 노출
- 다른 사장이 고객 예약을 취소 또는 복구할 수 있음
- 소유권 경계를 위반하는 기능·보안 결함

#### 권장 수정

- **기획서 변경 없음**
- **설계서 변경 없음:** 기획서의 우편번호 기준 소유권 규칙으로 충분하다.
- **구현물:** 호텔명 일치 조건을 제거한다. C-01의 내부 식별자를 도입하면 장기적으로 `hotelId` 기준으로 판정한다.

### C-03. 체크인·체크아웃 예정일을 지나면 상태 전이가 영구 차단됨

**판정: 치명적, 기획서 내부 로직 공백 및 문서 충돌**

#### 근거

- 기획서 6.3.2는 체크인과 체크아웃 시 현재 날짜가 각각 예정 날짜와 같아야 한다고 규정한다.
- 설계서 9.1과 9.2는 현재 기준 시각이 예정 시각 이후이면 처리 가능하다고 규정하며 같은 날짜일 것을 요구하지 않는다.
- 구현은 기획서처럼 정확히 같은 날짜만 허용한다.
  - `src/main/java/service/GuestService.java:230-241`
- 기준 시간은 관리자가 미래로 크게 진행할 수 있다.
- 상태 모델에는 `NO_SHOW`, 강제 체크아웃, 자동 종료 상태가 없다.

#### 모순 발생 시나리오

1. 예약 A의 체크인은 6월 10일 15:00이다.
2. 관리자가 기준 시간을 6월 11일 00:00으로 변경한다.
3. A는 여전히 `RESERVED`지만 체크인 날짜가 지났으므로 체크인할 수 없다.
4. 취소와 예약 변경도 이미 체크인 시각이 지났으므로 불가능하다.
5. A를 종료 상태로 옮길 수 있는 경로가 없다.

`CHECKED_IN` 상태에서 체크아웃 날짜를 지나도 동일하게 영구 정체된다. 이때 실제로 계속 투숙 중인 상태처럼 보이지만 중복 검사는 예약에 기록된 과거 숙박 구간만 사용하므로 이후 예약을 막지 못한다.

#### 영향

- 영구적으로 처리 불가능한 예약 레코드 발생
- 실제 투숙 상태와 예약 가능 여부가 불일치
- 평점 작성 및 월별 평점 집계까지 진행 불가

#### 권장 수정

- **기획서 최소 보완:** 예정 시각 이후 늦은 체크인·체크아웃을 허용할지, 예정일 경과 시 자동 상태 전이를 수행할지 하나를 명시한다.
- 최소 변경 권고안은 설계서의 규칙처럼 `현재 기준 시각 >= 예정 시각`이면 늦은 체크인·체크아웃을 허용하는 것이다. 기존 지각 패널티 규칙과도 자연스럽게 연결된다.
- **설계서:** 놓친 날짜 및 큰 폭의 기준 시간 변경 시 상태 전이를 명시한다.
- **구현물:** 날짜 동일 조건을 제거하고 상태·시각 조건으로 판단한다.

### C-04. 취소 승인 마감 후 `CANCEL_PENDING` 재생성 가능

**판정: 높음, 상태 전이 모순**

#### 근거

- 기획서 5.3.7과 설계서 7.2는 체크인 24시간 전까지 미처리된 `CANCEL_PENDING` 예약을 `RESERVED`로 복구하도록 규정한다.
- 구현도 자동 복구한다.
  - `src/main/java/service/GuestService.java:313-330`
- 그러나 취소 가능 조건은 현재 시각이 체크인 시각 전인지 여부만 검사한다.
  - `src/main/java/service/GuestService.java:220-223`
- 체크인 날짜까지 3일 이하이면 다시 `CANCEL_PENDING`으로 만든다.
  - `src/main/java/service/GuestService.java:249-272`

#### 모순 발생 시나리오

1. 체크인 시각이 6월 10일 15:00인 예약이 있다.

5. 재실행하면 미래 기준 시간에서 일부 자동 처리만 반영된 상태로 시작한다.

#### 영향

- 시간은 진행됐지만 해당 시간에 필요한 상태 전이는 누락됨
- 같은 명령을 다시 실행하기 어려움
- 오류 메시지가 실제 데이터 저장 실패 원인을 숨김

#### 권장 수정

- **기획서·설계서 변경 없음**
- **구현물:** 새 기준 시간과 모든 자동 처리 결과를 메모리 스냅샷에서 계산·검증한 뒤 C-10의 원자적 저장으로 한 번에 반영한다.
- 입력 오류와 데이터 저장·무결성 오류를 구분하여 안내한다.

## 6. 추가 확인된 모순 및 위험

### A-01. 호텔 우편번호 연쇄 변경은 스냅샷 문서를 위반하지만 참조 무결성에는 필요함

- 현재 구현처럼 예약 우편번호를 변경하면 과거 위치 스냅샷이 사라진다.

### A-03. 저장 실패 전에 메모리 객체를 직접 변경하여 재시도도 안전하지 않음

- 우편번호 변경과 예약 변경 등은 저장 전에 기존 객체를 직접 수정한다.
- 파일 저장이 실패해도 메모리 객체는 이미 변경된 상태로 남는다.
- 단순히 파일 저장만 원자화하지 말고, 작업 전 메모리 스냅샷 또는 복사본에서 검증해야 한다.

### A-04. 체크인·체크아웃 지각 패널티와 정확한 날짜 제한이 서로 충돌

- 지각 패널티는 예정 시각보다 1시간 이상 늦을 때 부여하도록 설계되어 있다.
- 기획서와 구현의 정확한 날짜 조건 때문에 자정을 넘긴 지각 체크인·체크아웃은 아예 처리할 수 없어 패널티도 부여할 수 없다.
- C-03의 늦은 처리 정책으로 함께 해결해야 한다.

### A-05. 달력 차단 대상 상태가 문서마다 다름

- 기획서 5.3.2는 `CANCELLED`가 아닌 모든 예약을 실제 시간 중복 검사 대상으로 규정한다.
- 기획서 6.3.1의 달력 설명은 `RESERVED`, `CANCEL_PENDING`만 차단하는 것으로 서술한다.
- 설계서와 구현은 `CANCELLED`를 제외한 모든 상태를 차단 예약으로 본다.
- `CHECKED_OUT`의 과거 숙박 구간을 이력 보존 차원에서 중복 검사 대상으로 둘지는 타당하지만, 달력 표시 규칙과 최종 중복 규칙은 구분해 명시해야 한다.

### A-06. 평점 문법이 기획서 내부와 설계서에서 일관되지 않음

- 기획서 4.12는 소수점 1자리 형식을 요구하지만 일부 사용자 흐름 예시는 정수 평점을 사용한다.
- 설계서 9.3과 10은 0.0~10.0 유리수를 허용하고 0.5 단위 제한을 두지 않는다고 설명한다.
- 구현은 정수 또는 소수점 한 자리만 허용한다.
- 최소 변경 권고안은 기획서 4.12의 “소수점 한 자리”를 기준으로 예시와 설계서를 정리하는 것이다.

### A-07. 개행 정의와 구현의 마지막 개행 검사가 다름

- 기획서는 LF, CR, CRLF 중 하나 또는 조합을 개행으로 정의한다.
- 구현은 파일 마지막 바이트가 LF가 아니면 오류로 처리한다.
  - `src/main/java/persistence/FlatFileStore.java:487-490`
- 따라서 CR만 사용하는 파일은 기획서상 유효하지만 구현에서는 거부된다.
- 저장 파일의 허용 개행을 LF/CRLF로 좁혀 문서화하거나 구현이 CR도 허용하도록 해야 한다.

## 7. 권장 해결 순서

### 1단계: 기획서 최소 결정

다음 네 항목만 기획서에 최소 보완한다.

1. 호텔·객실의 변경 불가능한 내부 식별자와 표시용 스냅샷을 구분한다.
2. 예정일을 지난 체크인·체크아웃을 허용할지 또는 자동 처리할지 정한다.
3. 패널티 종료일의 포함 여부와 정확한 N일 계산 방식을 정한다.
4. 자동 영업정지 취소 원인을 영구 보존할 공식 필드를 정한다.

페널티·영업정지 컬럼 오기 등 객관적인 오류는 정책 변경이 아니라 문구 정정으로 처리한다.

### 2단계: 설계서 정합성 정리

- 내부 식별자와 스냅샷의 책임 분리
- 상태 전이 표에 승인 마감 이후 재취소 요청과 예정일 경과 처리 추가
- 월별 자동 작업을 모든 통과 월에 대해 시간순 실행하도록 명시
- 패널티 기간의 포함·배제 규칙 통일
- 달력 표시와 실제 예약 가능 검사의 차이 명시
- 저장 파일 canonical 파서와 UI 입력 파서 분리
- 다중 파일 원자적 저장 및 실패 복구 절차 추가
- 최종 파일별 컬럼 표 확정

### 3단계: 구현물 중심 수정

1. `hotelId`, `roomId` 또는 동등한 안정 키 도입
2. 사장 예약 소유권 판정에서 호텔명 조건 제거
3. 늦은 체크인·체크아웃과 취소 승인 마감 상태 전이 수정
4. 모든 미처리 월의 자동 작업을 순서대로 실행
5. 월별 작업 후 최종 만료 데이터 정리
6. 날짜 전체 차단과 실제 시간 중복 판정 정합화
7. 패널티 기간 계산 통일
8. 엄격한 파일 파서 분리
9. 다중 파일 원자적 저장 및 메모리 롤백 구현

## 9. 검사 한계 및 현재 상태

- 이번 결과는 PDF 원문과 Java 소스의 정적 대조를 중심으로 작성했다.
- 검사 보고서 PDF는 평가하지 않았다.
- 자동 테스트 프레임워크와 테스트 코드는 저장소에서 확인되지 않았다.
- 현재 Java 소스는 컴파일 가능한 상태지만, 정상 컴파일은 위 로직 모순이 없음을 의미하지 않는다.
- 실제 프로그램 실행은 시작 과정에서 데이터 파일을 정규화·저장할 수 있으므로, 원본 데이터 변경을 피하기 위해 이번 검사에서는 수행하지 않았다.
