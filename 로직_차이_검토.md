# 기획서·설계서 대비 로직 차이 검토

이 문서의 항목은 단순 문구 수정 범위를 넘으므로 코드에 반영하지 않았다.

## 문서 간 기준 결정 필요

### 1. 예약 기간과 체크아웃 날짜

- 기획서 6.3.1은 예약 날짜 하나와 체크인·체크아웃 시간만 입력받는다.
- 설계서 4.2, 7.2는 체크아웃 날짜를 자동으로 익일로 결정하지 않는다고 명시한다.
- 현재 구현은 체크아웃 날짜를 체크인 날짜의 다음 날로 자동 결정하고, 무결성 검사도 1박 2일만 허용한다.
- 다박 예약 허용 여부와 체크아웃 날짜 입력 방식의 기준 결정이 필요하다.
- **[결정사항]** 1박 2일 예약만 허용한다. 체크아웃 날짜는 체크인 날짜의 다음 날(익일)로 자동 결정하며, 무결성 검증도 1박 2일만 허용하는 현재 구현 방식을 유지한다.

### 2. 호텔별 페널티의 적용 범위

- 설계서 3.2, 7.2는 전역 페널티는 전체 호텔, 호텔별 페널티는 해당 호텔 예약만 제한한다고 설명한다.
- 기획서 6.3.1, 6.3.2는 종류와 무관하게 활성 페널티가 하나라도 있으면 예약·예약 변경·예약 취소를 막는다.
- 현재 구현은 기획서 프롬프트 흐름처럼 활성 페널티가 하나라도 있으면 모든 예약 관련 기능을 막는다.
- 호텔별 페널티를 해당 호텔에만 적용할지 결정이 필요하다.
- **[결정사항]** 호텔별 페널티는 해당 호텔(해당 우편번호)의 예약·변경·취소만 제한하고 다른 호텔은 허용하도록 변경한다. 전역 페널티(GLOBAL 우편번호)인 경우에만 전체 호텔에 대해 예약을 제한한다.

### 3. 달력에서 차단할 예약 상태

- 기획서 6.3.1은 `RESERVED`, `CANCEL_PENDING` 상태의 날짜를 차단한다고 설명한다.
- 설계서 4.3, 12장은 `CANCELLED`를 제외한 모든 상태를 차단 예약으로 본다.
- 현재 구현은 설계서 기준으로 `CHECKED_IN`, `CHECKED_OUT`도 차단한다.
- **[결정사항]** `CANCELLED`를 제외한 모든 상태(`RESERVED`, `CANCEL_PENDING`, `CHECKED_IN`, `CHECKED_OUT`)를 차단 예약으로 취급하여 중복 예약을 완전히 방지하는 현재 구현 방식을 유지한다.

### 4. 입실·퇴실 가능 날짜

- 기획서 6.3.2는 현재 날짜가 각각 체크인 날짜, 체크아웃 날짜와 같아야 한다고 설명한다.
- 설계서 9.1, 9.2는 현재 기준 시각이 예약 시각 이후인지만 조건으로 둔다.
- 현재 구현은 설계서 기준으로 날짜가 지난 뒤에도 입실·퇴실 처리가 가능하다.
- **[결정사항]** 기획서 기준을 따라 현재 기준 날짜(baselineDate)가 반드시 예약의 체크인 날짜(입실 시) 혹은 체크아웃 날짜(퇴실 시)와 일치해야만 처리가 가능하도록 변경한다. 당일이 지나면 입·퇴실 처리가 불가하다.

### 5. 평점 입력 형식

- 기획서 4.12는 소수점 한 자리 형식을 명시한다.
- 설계서 10장은 0 이상 10 이하의 유리수를 허용하고 0.5 단위 제한을 두지 않는다.
- 현재 구현은 `Double.parseDouble`로 해석 가능한 0 이상 10 이하 값을 허용한다.
- **[결정사항]** 기획서 기준을 따라 소수점 이하 첫째 자리까지만 입력을 허용하도록 변경한다. 소수점 둘째 자리 이하의 값이 입력될 경우 문법 오류 처리한다. (0.0 ~ 10.0 범위의 소수점 첫째 자리 값만 허용)

### 6. 데이터 파일 문법 표기

- 기획서 5.1의 페널티·영업정지 레코드 필드 수 설명은 기획서의 다른 절 및 설계서 3장과 서로 다르다.
- 현재 구현은 설계서 기준인 `penalty.txt` 3필드, `suspension.txt` 4필드를 사용한다.
- **[결정사항]** 보류 (논의 예정)

### 7. 예약 스냅샷 참조

- 기획서 5.2 앞부분은 예약의 업소·방 정보가 현재 파일에 존재해야 한다고 설명한다.
- 같은 절의 예외와 설계서 4.3, 4.5는 변경 전 예약 정보를 스냅샷으로 허용한다.
- 현재 구현은 스냅샷 기준으로 현재 업소·방과 달라도 무결성 오류로 처리하지 않는다.
- **[결정사항]** 보류 (논의 예정)

## 현재 구현과 문서의 차이

### 1. 복귀·종료 명령어 처리 범위

- 기획서는 다수 메뉴와 입력 단계에서 `;` 복귀와 `;;` 종료를 허용한다.
- 현재 구현은 `;;`를 초기 메뉴에서만 종료 명령으로 처리하며, `;`도 화면마다 동작이 다르다.
- 전역 명령 처리 방식과 입력 중 파일 변경 취소 정책을 함께 설계해야 한다.
- **[결정사항]** 보류 (논의 예정)

### 2. 보유 업소 유무에 따른 사장 메뉴

- 기획서 6.2.2는 보유 업소가 없으면 `업소 등록`, `로그아웃`만 표시하도록 한다.
- 현재 구현은 보유 업소 유무와 관계없이 항상 4개 메뉴를 표시한다.
- **[결정사항]** 기획서 기준을 따라 등록된 보유 업소가 없는 경우에는 메뉴에 `업소 등록`과 `로그아웃`만 표시하도록 수정한다. 1개 이상의 업소를 보유하고 있을 때만 전체 메뉴(업소 등록, 업소 정보 수정, 예약자 관리, 로그아웃)를 노출한다.

### 3. 여러 달을 건너뛴 기준 시간 변경

- 기획서 5.3.5는 변경 전·후 기준 시간 사이에 포함된 모든 월별 점검 시점을 처리하도록 한다.
- 현재 `MonthlySystemService.runIfNeeded`는 새 기준 시간의 직전 한 달만 처리한다.
- 기준 시간을 여러 달 앞으로 변경하면 중간 월의 평균 평점 점검이 누락될 수 있다.
- **[결정사항]** 현재 구현 방식을 유지하여 변경된 새 기준 시각의 직전 한 달(이전 달)에 대해서만 평점 점검을 실행하고 중간 월들은 건너뛴다.

### 4. 객실 기본 시간 무결성

- 설계서 4.2는 객실 기본 체크인 시간과 체크아웃 시간이 같거나 체크인이 더 늦은 경우를 허용하지 않는다.
- 현재 파일 무결성 검사, 업소 등록, 방 정보 수정에서는 두 시간의 선후 관계를 검사하지 않는다.
- **[결정사항]** 보류 (논의 예정)

### 5. 숫자 상한 경계값

- 기획서 4.9는 방 번호를 10,000 미만, 4.10은 인원을 1,000 미만으로 정의한다.
- 현재 파서는 각각 10,000과 1,000을 허용한다.
- **[결정사항]** 기획서 기준을 따라 방 번호는 10,000 미만(즉, 1~9,999), 수용 인원은 1,000 미만(즉, 1~999)만 허용하도록 상한 경계값 검증 로직을 변경한다.

### 6. 데이터 무결성 누락 항목

- 기획서 5.2의 `예약 생성 날짜보다 체크인 날짜가 이후` 조건을 파일 로드 시 검사하지 않는다.
- 기획서 5.2의 영업정지 사유가 비어 있지 않아야 한다는 조건을 검사하지 않는다.
- `time.txt`, `system.txt`가 여러 줄이어도 현재 구현은 줄을 합쳐 하나의 값처럼 처리할 수 있다.
- 기획서가 요구하는 데이터 파일 마지막 행의 개행 여부를 검사하지 않는다.
- **[결정사항]** 4가지 누락 항목을 모두 추가 적용한다:
  1. 예약 로드 시 예약 생성 날짜보다 체크인 날짜가 이후(체크인 날짜가 생성 날짜보다 뒤)인지 검사 추가.
  2. 영업정지 데이터 로드 시 영업정지 사유가 비어 있지 않은지 검사 추가.
  3. `time.txt`와 `system.txt` 파일 로드 시 2줄 이상인 경우 문법 오류 처리 추가.
  4. 모든 데이터 파일 로드 시 마지막 행에 개행이 없는 경우 문법 오류 처리 추가.

### 7. 잘못된 `time.txt` 경고

- 기획서 5.3.4는 유효하지 않은 `time.txt`를 KST 현재 시각으로 보정하기 전에 경고 메시지를 출력하도록 한다.
- 현재 구현은 보정만 하고 경고를 출력하지 않는다.
- **[결정사항]** 기획서 기준을 따라 유효하지 않은 `time.txt` 값을 보정하기 전, 경고 메시지를 출력하도록 로직을 변경한다.

### 8. 입력 단계별 검증 시점과 오류 구분

- 기획서 6.3.1은 예약 체크인·체크아웃 시간이 객실 허용 범위를 벗어나면 해당 입력 단계에서 다시 입력받도록 한다.
- 현재 구현은 두 시간을 모두 받은 뒤 예약 확정 단계에서 범위를 검사한다.
- 사장 방 번호·인원·시간 입력도 문법 오류와 의미 오류에 따라 다른 문구를 요구하지만 현재는 같은 문구로 처리한다.
- **[결정사항]** 기획서 기준을 적용한다:
  1. 고객 예약 시 체크인 및 체크아웃 시간이 객실 허용 범위를 벗어날 경우 해당 입력 단계에서 즉시 검사하여 잘못된 값일 경우 재입력을 받는다.
  2. 사장의 방 번호, 수용 인원, 시간 입력 등 모든 사용자 입력 시점에 문자열 포맷 불일치("문법 오류")와 논리적 조건 불일치("의미 오류")를 구분하여 기획서가 명시한 서로 다른 오류 문구를 출력한다.

### 9. 자동 영업정지 취소의 최근 취소 횟수 제외

- 문서는 자동 영업정지로 취소된 예약을 고객의 최근 취소 횟수에서 항상 제외하도록 한다.
- 현재 구현은 해당 영업정지 레코드가 남아 있을 때만 자동 취소 여부를 판별한다.
- 영업정지 레코드가 만료되어 삭제된 뒤에는 과거 자동 취소 예약이 고객 취소로 계산될 수 있다.
- **[결정사항]** 기획서 기준을 충족하기 위해, 영업정지 만료 여부와 상관없이 자동 취소된 예약은 항상 고객 취소 횟수 산정에서 배제될 수 있도록 예약 모델 혹은 판별 기준에 영구 플래그나 별도 표식을 지정하여 영구적으로 제외하도록 개선한다.

### 10. 호텔 우편번호 변경과 참조 데이터

- 현재 우편번호 변경은 `hotel.txt`, `room.txt`만 갱신한다.
- 기존 호텔별 페널티와 영업정지는 이전 우편번호를 유지해 다음 실행 시 참조 무결성 오류가 발생할 수 있다.
- 기존 예약은 스냅샷 우편번호를 유지하므로 예약 변경 시 현재 호텔을 찾지 못할 수 있다.
- 호텔명과 우편번호를 모두 변경하면 사장의 기존 예약 조회에서 해당 예약이 누락될 수 있다.
- **[결정사항]** 호텔 우편번호 변경 시 기존의 연관 데이터(`room.txt`뿐만 아니라 예약 `reservation.txt`, 페널티 `penalty.txt`, 영업정지 `suspension.txt` 등) 내의 우편번호 정보도 일괄적으로 새 우편번호로 변경하는 연쇄 갱신(Cascade Update)을 구현함으로써, 데이터 연결 관계를 보존하고 재기동 시 무결성 오류가 생기지 않도록 차단한다.
